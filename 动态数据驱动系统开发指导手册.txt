”太平镇教育人事管理系统“动态数据驱动系统开发指导手册（本地开发版）‍

第一章：核心理念与系统架构

1.1 核心设计哲学：配置驱动与元编程
本系统的核心思想是将系统的“业务逻辑”与“技术实现”彻底解耦。其所有行为、外观和数据处理流程均由外部的配置文件定义，从而实现真正的“动态数据驱动” 。这意味着系统的变化和扩展主要通过修改配置文件来实现，而非重写代码。

1.2 系统生命体全景图
我们将系统比作一个生命体，其各部分功能如下：

骨架：框架，即唯一技术栈（Python FastAPI + Vue.js 3 + PostgreSQL + SQLAlchemy + Element Plus + ECharts）。
血肉：四大核心配置文件，定义了系统的具体形态和功能 。
神经：事件总线，负责传递信息和指令，确保系统各部分协调同步。
大脑：元数据引擎，负责解析所有配置文件，形成系统认知，并指挥各部分工作 。
分析器官：报表生成引擎，负责深度思考和战略分析，将数据转化为洞察  。
表达器官：可视化配置与渲染引擎，负责将分析结果以图表等形式直观呈现。
1.3 唯一技术选型
为确保系统的强健性与稳定性，我们指定以下唯一技术栈：

后端：Python FastAPI
前端：Vue.js 3 (Composition API)
数据库：PostgreSQL（本地安装）
ORM：SQLAlchemy 2.0
前端UI库：Element Plus
前端图表库：Apache ECharts

第二章：核心配置文件体系
配置文件是系统的“血肉”，是驱动一切行为的规则基础 。

navigation.json：定义系统的导航结构，即“行动地图”。可通过“模块管理组件”进行图形化动态管理 。系统全部模块均由此配置文件统一管理。新增主模块、子模块、节点均由此配置管理文件自动生成路径、路由。系统路径、路由均由此文件集中统一管理。导航菜单初始时只有一个“系统管理”主模块，主模块下只有一个模块管理功能子模块。其他禁止预设。
field_mappings.json：1.定义数据库的“设计蓝图”，即每个数据表的详细结构（字段名、数据类型、主键、外键等）  。2.定义文件上传时的“翻译词典”与智能填充规则，即原始中文表名与系统内部标准英文表名的对应关系 ,原始中文字段与系统内部标准英文字段的对应关系 。3.为每个字段增加一个 source_field_name 或类似的属性，用于记录上传文件中的原始中文名。必须保证中文表名、中文字段名与对应的映射英文表名、英文字段名的唯一性，一个中文表名（字段名）与唯一一个英文映射表名（英文映射字段名唯一成对关系），防止一个中文表名映射多个英文表名，防止一个中文字段名对应多个英文字段名，防止一个英文表名对应多个中文表名，防止一个英文字段名对应多个中文字段名。4.翻译时要全文翻译，精准翻译，禁止用顺序码、哈希值等无意义的形式翻译，要逐个字段翻译，特别大型数据表导入时，有多个字段，要逐个精准翻译。
ui_components.json：定义前端界面的“装修方案”，即表单、列表等UI组件的结构。UI选择有针对性表义的形象生动的图标。
report_definitions.json：定义报表的“配方集合”，即每个报表的完整生成规则 。



第三章：核心业务流程

3.1 动态数据导入流程（一次性数据清洗与导入）‍
此流程是一个必须保持完整性的原子事务 。

文件上传与预解析：用户上传文件（如Excel），直接提取中文表名，系统提取元数据和字段结构（中文字段名列表）。
智能匹配与人工审核：
系统调用后端API，将上传文件的中文表名、中文字段列表与field_mappings.json中已有的配置进行比对。已有表名、字段直接引用其英文字段名；未匹配表名、字段自动生成唯一且符合规范的英文表名、英文字段名，并原子性地保存到配置中 。
用户在“数据预览与清洗工作台”中对智能推断的字段属性（数据类型、约束）和映射关系进行审核与修正，完成字段属性配置并以当前文件名+YYYY-MM-DD格式进行命名保存，当再次导入相同文件时复用。
一次性数据清洗与更正：
系统根据最终确认的字段属性对所有数据进行验证 。
数据被分为“全部数据”、“合格数据”和“错误数据”。用户仅对“错误数据”进行行内和弹窗内编辑更正，合格后自动移入“合格数据”列表 ，点击错误数据时增加一个错误数据导出按钮，按当前文件名+YYYY-MM-DD格对导出文件进行命名。
一次性导入与系统更新：
用户确认后，系统将所有“合格数据”一次性导入数据库。不合格数据直接按当前文件名+YYYY-MM-DD格对导出文件进行命名并导出，同时给出导出报告。参考：D:\erp_thirteen\学历类型字典_2026-01-29 .txt
关键原子操作：在同一事务中，动态创建或更新数据表结构，并同步更新table_schemas.json和navigation.json等配置文件  。
通过事件总线发布更新事件，通知前端导航树等组件刷新 。
3.2 关系型数据关联模型

主表与字典表设计：所有核心实体表（如employees）和字典表（如education_type_dict）必须使用系统自增的“代理主键”（如id）作为唯一标识，以确保稳定性和关联效率 。本系统以教师基础信息表为主表，子表即要关联系统主表，也要关联一个或多个字典表。一定要在前期设计自动关联功能。
外键关联：子表（如teacher_education_records）通过外键字段（如teacher_id, education_type_id）关联到对应主表或字典表的主键，建立精确的数据关系 。
UI自动关联：在数据录入界面，对于外键字段，应渲染下拉选择组件，其选项列表通过查询相关字典表动态获取，确保数据关联的准确性与用户操作的便捷性。
3.3 动态报表生成流程

设计报表：用户在“报表设计器”中通过拖拽字段定义报表，配置被保存到report_definitions.json 。
生成查询：用户请求报表时，“报表生成引擎”读取配置，动态生成包含必要JOIN的SQL查询，以关联主表、子表和字典表数据 。
渲染图表：数据返回后，“渲染引擎”调用ECharts将数据可视化。

第四章：开发、协作与调试方法论

4.1 与AI代码助手的精准协作模式

战略预备：先将本指导手册完整交给AI，建立其全局认知 。
战术执行——精准指令三要素：
划定精确范围：明确指令修改的具体文件、函数，并声明禁止修改的关联区域。
声明核心约束：强调不可更改的系统架构契约，如“必须原子性更新navigation.json”、“字段映射必须源自field_mappings.json”  。
要求影响分析：指令AI分析修改对核心配置文件及其他模块的潜在影响。
4.2 关键调试与问题解决指南

‍“导入失败”排查铁律：
首要步骤：检查并增强后端导入API的异常处理。必须捕获所有异常，并将详细错误信息（e.message）结构化后返回给前端，绝对禁止返回非JSON格式的笼统提示 。
验证点：检查服务器上table_schemas.json、navigation.json等配置文件的格式有效性与写入权限 。
数据验证：核对导入数据的值是否与table_schemas.json中定义的字段属性（如唯一性约束、非空约束）冲突 。
避免“修甲坏乙”‍：任何修改都必须考虑其对依赖同一配置文件的其他模块的影响，修改后需进行关联功能回归测试 。
4.3 本地开发与部署

开发模式：采用“本地开发，服务器部署”。在本地安装PostgreSQL、Node.js和Python环境进行编码 。
服务器：推荐使用Ubuntu Server 22.04 LTS作为生产环境操作系统 。

细粒度动态权限控制：权限控制到数据行和字段级别。
操作日志与审计追踪：记录所有关键操作，提供审计报表。

策略驱动：提供图形界面配置备份计划（对象、周期、类型、保留策略）。
自动化执行：后台服务自动执行备份、压缩、加密和存储。
监控与告警：监控备份状态，失败时自动通知。
