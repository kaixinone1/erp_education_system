# 太平教育人事工资党建管理系统 - 开发原则（最高准则）

**创建时间：2026-01-08**
**作用：本文档是开发本系统的最高准则，所有开发工作必须遵循**

---

## 一、系统定位

### 1.1 系统本质
**系统是一个有生命力的有机体，不是功能的堆砌。**

```
┌─────────────────────────────────────────────────────────────────┐
│                    系统生命体模型                               │
├─────────────────────────────────────────────────────────────────┤
│  骨架：技术栈（FastAPI + Vue 3 + PostgreSQL + SQLAlchemy）     │
│  血肉：四大核心配置文件（定义系统形态和功能）                     │
│  神经：事件总线（传递信息和指令，协调各部分）                     │
│  大脑：元数据引擎（解析配置，形成系统认知，指挥工作）             │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 核心设计哲学
**配置驱动与元编程：业务逻辑与技术实现彻底解耦**

- 所有行为由配置文件定义
- 修改行为只需修改配置，无需改代码
- 系统扩展通过配置实现，不是写新代码

## 二、开发宪法（五条铁律）

**任何时候感到迷茫，回到这五条原则**

### 2.1 零修改原则
**禁止抄袭网上代码，要原创、有生命力**

| 错误做法 | 正确做法 |
|----------|----------|
| 网上找代码复制粘贴 | 根据系统需求原创设计 |
| 找开源项目抄一改 | 代码融入现有架构 |
| 简单拼凑碎片化 | 代码有内在一致性 |

### 2.2 蓝图先行原则
**编写任何代码前，必须先提交功能蓝图**

- 包含：文件结构、API设计、数据流图
- 待批准后方可编码
- 防止盲目开发，减少返工

### 2.3 配置驱动原则
**所有新功能的行为、外观、数据结构由外部配置定义**

| 层面 | 配置方式 |
|------|----------|
| 导航结构 | `navigation.json` |
| 数据库结构与字段映射  | `field_mappings.json` |
| 界面组件 | `ui_components.json` |
| 报表规则 | `report_definitions.json` |

### 2.4 原子交付原则
**以功能模块为单位，一次性完成交付**

- 一个功能模块的所有相关代码必须同时完成
- 避免半成品，保证质量
- 便于测试和验收

### 2.5 测试隔离原则
**禁止使用生产环境真实数据测试**

- 创建独立的虚构测试数据
- 测试后清理测试数据
- 保护生产数据安全

## 三、表名映射规则

**这是用户最关心的问题，必须严格遵循**

### 3.1 核心原则：一 一对应

```
┌─────────────────────────────────────────────────────────────────┐
│                    表名映射系统                                  │
│                                                                 │
│  用户看到           映射关系             数据库存储              │
│  ┌─────────┐      ┌─────────────┐      ┌─────────────────┐     │
│  │中文表名  │ ──→  │ raw_table_  │ ──→  │ system_table_   │     │
│  │         │      │   name      │      │     name        │     │
│  └─────────┘      └─────────────┘      └─────────────────┘     │
│                                                                 │
│  字段名同理：                                                  │
│  ┌─────────┐      ┌─────────────┐      ┌─────────────────┐     │
│  │中文字段名│ ──→  │ raw_field_  │ ──→  │ system_field_   │     │
│  │         │      │   name      │      │     name        │     │
│  └─────────┘      └─────────────┘      └─────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 映射表结构

**数据库表：`global_table_mapping_registry`**

| 字段 | 说明 | 示例 |
|------|------|------|
| `raw_table_name` | 中文表名（唯一） | 个人身份字典 |
| `system_table_name` | 英文表名 | personal_identity_types |
| `module` | 所属模块 | TeacherManagement |
| `sub_module` | 所属子模块 | null |

### 3.3 操作规则

| 操作 | 规则 |
|------|------|
| **第一次导入** | 生成英文表名，写入映射表 |
| **后续导入** | 直接查表，不再重新生成 |
| **修改英文表名** | 用户可修改，更新映射表 |
| **查询中文表名** | 从映射表查询，不动态转换 |

### 3.4 模块前缀规则

| 模块 | 前缀 | 示例 |
|------|------|------|
| 人事管理 | `hr_` | `hr_teacher_basic_info` |
| 工资管理 | `salary_` | `salary_payment_records` |
| 党建管理 | `party_` | `party_member_info` |
| 学生管理 | `student_` | `student_basic_info` |
| 系统管理 | `system_` | `system_users` |

## 四、数据流向规则

### 4.1 语言使用

| 数据流向 | 语言 | 原因 |
|----------|------|------|
| 用户 → 网页 | 中文 | 用户友好 |
| 网页 → 用户 | 中文 | 用户友好 |
| 前端 → 后端 | 英文 | 统一标准 |
| 后端 → 前端 | 英文/中文 | 前端需要中文显示 |
| 数据库存储 | 英文 | 行业规范 |

### 4.2 转换规则

```
用户上传文件
    │
    ▼
┌─────────────────────────────┐
│ 解析文件，获取中文表名        │
│ 和中文字段名                 │
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│ 查询 global_table_mapping   │
│                              │
│ 如果不存在：                 │
│   - 生成英文表名/字段名       │
│   - 写入映射表               │
│                              │
│ 如果已存在：                 │
│   - 直接使用已有映射         │
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│ 返回英文表名给后端           │
│ 返回中文字段名给前端         │
└─────────────────────────────┘
```

## 七、与用户协作模式

### 7.1 角色定位

| 角色 | 职责 |
|------|------|
| **用户** | 总设计师，发布指令，审核蓝图，验收成果 |
| **AI** | 执行者，接收指令，提交蓝图，编码交付 |

### 7.2 工作流程

```
1. 用户发布指令
      │
      ▼
2. AI理解需求，提交功能蓝图
      │
      ▼
3. 用户审核蓝图（批准或要求修改）
      │
      ▼
4. AI编码实现（原子交付）
      │
      ▼
5. 用户验收（通过或要求修改）
      │
      ▼
循环直到完成
```

### 7.3 沟通原则

| 原则 | 说明 |
|------|------|
| **小步快跑** | 将复杂功能拆解为小指令 |
| **持续反馈** | 每步完成后立即汇报 |
| **明确指令** | 用户指令要清晰具体 |
| **先蓝图后编码** | 没有蓝图不开始编码 |

## 八、开发检查清单

**每次开发前，检查以下事项：**

### 8.1 代码检查
- [ ] 是否遵循零修改原则（原创、不抄代码）？

## 九、常见问题处理

### 9.1 表名冲突
**问题**：导入文件时，表名已存在
**处理**：
1. 查询 `global_table_mapping_registry`
2. 如果存在，返回已有映射
3. 如果不存在，生成新映射并写入

### 9.2 字段名冲突
**问题**：同一表多次导入，字段名重复
**处理**：
1. 查询 `global_field_mapping_registry`
2. 如果存在，返回已有映射
3. 如果不存在，创建新映射

### 9.3 配置不一致
**问题**：配置文件与数据库不一致
**处理**：
1. 以数据库为准
2. 更新配置文件与数据库同步
3. 后续操作从数据库获取映射

### 9.4 性能问题
**问题**：大量数据导入性能差
**处理**：
1. 使用批量插入
2. 使用事务保证原子性
3. 使用事件总线异步通知
