## 执行计划

### 指令1：确立配置驱动原则，审查并清除硬编码

**发现的问题：**
1. `validation_service.py` 中 Level2Validator 和 Level3Validator 有大量硬编码：
   - `target_field == 'age'` 或 `'年龄' in field_name`
   - `target_field == 'id_card'` 或 `'身份证' in field_name`
   - `target_field == 'gender'` 或 `'性别' in field_name`
   - 学历、政治面貌等固定值列表

2. 验证器应该从 `merged_schema_mappings.json` 读取验证规则，而不是硬编码

**修改方案：**
- 修改 `Level1Validator`：从配置文件读取数据类型、必填、长度等规则
- 修改 `Level2Validator`：从配置文件读取业务验证规则（正则、范围等）
- 修改 `Level3Validator`：从配置文件读取关联验证规则
- 修改 `Level4Validator`：从配置文件读取外键关联规则

---

### 指令2：重构"字段属性配置"步骤，实现与主表配置的关联

**修改文件：**
1. `FieldConfigPanel.vue` - 添加关联类型选择器
2. `field_config_routes.py` - 添加获取主表列表接口
3. `merged_schema_mappings.json` - 添加 relation 配置结构

**功能：**
- 字段配置时可以选择"关联类型"：外键/字典/无
- 选择外键时，动态加载主表列表
- 选择目标表和目标字段
- 保存 relation 配置到 mappings

---

### 指令3：强化三级验证引擎，使其完全依赖字段配置

**修改文件：**
1. `validation_service.py` - 重写所有验证器
2. 添加配置加载器 - 从 `merged_schema_mappings.json` 读取规则

**验证规则来源：**
- Level 1：数据类型、必填、长度 → 从 `tables[table_name].fields`
- Level 2：正则、范围、唯一性 → 从 `mappings[field_name].validation`
- Level 3：关联存在性检查 → 从 `mappings[field_name].relation`

---

### 指令4：实现完全动态的原子化导入事务

**修改文件：**
1. `import_routes.py` - 重写 `/api/import/finalize` 接口
2. `universal_import_service_v3.py` - 已修改，需要微调
3. 添加原子事务包装器

**原子事务流程：**
```

  1. 动态创建/更新表结构
  2. 插入数据
```
